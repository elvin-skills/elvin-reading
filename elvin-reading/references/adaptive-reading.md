# 自适应阅读与真实反馈协议

## 目的

让下一次解释接住使用者上一轮的理解状态。反馈来自真实阅读对话，不要求使用者填写表格。

## 反馈识别

只记录使用者实际表达或可直接观察的行为。Agent 自己生成的提示、模板问题、未被回应的理解检查都不算反馈。

| signal | 常见表达 | 下一步 |
|---|---|---|
| `not_understood` | 还是没懂、句子连不起来 | 降低一级抽象度，换策略 |
| `too_abstract` | 太抽象、举个例子 | 使用同领域具体例子，再映射原句 |
| `too_detailed` | 太长、只说这里什么意思 | 改用最短语境义，停止扩展 |
| `understood` | 懂了、这样就清楚了 | 记录有效策略，立即回到阅读 |
| `wants_application` | 怎么用、对我有什么意义 | 增加应用场景或判断方法 |
| `wants_context` | 和前面有什么关系、作者为什么说它 | 补论证桥梁或最小背景 |
| `grammar_block` | 单词都懂但整句不懂 | 提取主干，再逐层装回修饰成分 |
| `vocabulary_block` | 这个词在这里什么意思 | 给当前语境义、搭配和记忆线索 |
| `ready_to_continue` | 可以了、继续 | 停止教学，不追加检查题 |

## 解释策略

`feedback` 命令中的 strategy 使用以下稳定值：

| strategy | 使用场景 | 输出动作 |
|---|---|---|
| `minimal-context` | 用户只想继续读 | 一句语境义 + 一个关键点 |
| `sentence-skeleton` | 主干关系不清 | 标出主语、谓语、宾语或补语 |
| `chunk-paraphrase` | 长句负荷过高 | 分意群解释，再合成自然中文 |
| `reference-substitution` | 代词、指代或省略导致困难 | 用实际指代对象替换后重读 |
| `logic-bridge` | 不理解论证跳跃 | 列证据、隐含前提和结论 |
| `minimal-background` | 缺领域前提 | 只补理解当前段落所需背景 |
| `domain-example` | 概念太抽象 | 使用同领域具体场景，再映射原句 |
| `old-example-contrast` | 旧材料有可靠记忆 | 并列旧例、新例、共同点和差异 |
| `application-case` | 用户追问应用 | 给判断条件、适用场景和边界 |

## 降阶顺序

同一问题解释失败时，每次只下降一级并更换方法：

1. `minimal-context`：当前句最短含义。
2. `sentence-skeleton`：只保留主干。
3. `chunk-paraphrase`：按意群拆开。
4. `reference-substitution`：替换指代、省略和抽象名词。
5. `domain-example`：用熟悉的同领域例子建立直觉。
6. 拆成更小的词汇、语法、背景或推理问题，并把当前答案标记为 `partial`。

不要连续增加更多定义。解释失败意味着当前表示方式不合适。

## 提升梯度

使用者连续理解同一种结构时：

1. 第一次完整解释机制。
2. 第二次调用旧例，只强调相同机制和本次差异。
3. 第三次先给识别提示，让使用者自行判断。
4. 使用者判断正确后，仅保留一句确认并继续阅读。

提升梯度的依据必须来自真实反馈或正确复述，不能只根据出现次数推断已经掌握。

## 反馈写入规则

在以下情形运行 `feedback`：

- 使用者明确说没懂、太抽象、太长或已经懂了；
- 使用者要求应用、背景、结构或更简单的解释；
- 一个理解检查得到明确回答；
- 同一障碍经过策略切换后出现可观察结果。

示例：

```bash
python3 scripts/elvin_reading.py feedback <项目目录> \
  --text "这样解释还是没懂" \
  --signal not_understood \
  --strategy sentence-skeleton \
  --outcome failed \
  --next-strategy chunk-paraphrase \
  --location "PDF page 12"
```

`outcome` 只能根据结果填写：

- `success`：使用者明确理解、正确复述或继续阅读；
- `partial`：解决一部分，仍有具体障碍；
- `failed`：使用者明确表示仍未理解；
- `unknown`：策略刚使用，尚未得到反应。

每条用户消息最多写一条反馈。反馈评价刚刚使用的策略；需要切换时把新方法写入同一记录的 `next_strategy`。不要把同一句“没懂”同时记成旧策略 `failed` 和新策略 `unknown`。新策略的效果由下一条真实用户反应决定。

词汇或语法的首次解释只生成 `partial` 阅读事件。收到 `success` 后，再用该 feedback_id 追加 `resolved` 事件；不要在用户确认前填写“已理解、已掌握”等 understanding。

## 回答长度

- 默认回答控制在解除当前障碍所需的最短长度。
- 用户说“继续”或表现出已理解后立刻停止扩展。
- 用户要求深入时再增加机制、背景、案例或迁移检验。
- 一次只问一个理解检查；词汇直译和明显简单问题通常不检查。
