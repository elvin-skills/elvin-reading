# 跨材料召回与复用规则

## 何时必须召回

当前材料不是项目中的第一份时，对下列请求先运行 `recall`：

- “这个词／术语是什么意思？”
- “这句话为什么这样写／主干是什么？”
- “作者这里为什么这样说？”
- “这个问题之前是不是遇到过？”
- 当前证据似乎回答了过去未解决的问题。

召回范围默认排除当前材料，确保“调用第一份记忆”不是把当前刚记录的内容当成旧记忆。
进入 Elvin-reading 阅读任务后，这是隐式召回：用户只需询问当前材料，不需要说“材料 A”
“旧笔记”“以前遇到过”或“帮我回忆”。Agent 根据当前词汇、结构和问题语义自行检索；
显式提到旧材料不能作为这项能力通过验收的前提。

## 相关性判断

脚本给的是候选，不是结论。复用前检查：

1. 类型是否相同或合理相邻，例如 QST 可以命中相关 CON，但不能只因共有常用词就关联。
2. 核心语义是否相同。词形相同但词义不同，应明确对比而不是复用旧释义。
3. 旧原句是否真的帮助理解当前句。
4. 旧解释是否仍可靠；新材料若修正旧解释，应记录 `revises`。

## 四段式对照

有效复用至少包含：

1. **材料 A／B 等明确来源**：面向用户显示材料标签、真实标题、具体位置、完整最小原句和当时解释。
2. **当前材料**：用对应材料标签显示当前完整原句及当前含义。
3. **共同点**：共享的核心词义、结构机制或问题框架。
4. **差异**：搭配、语气、逻辑角色、适用条件或证据强度的变化。

最后给一个“下次看到什么信号就能认出它”的记忆线索。仅提醒“以前见过”不算成功复用。
不得只展示旧句中的目标短语；旧材料标题、具体位置或完整最小原句缺少任意一项，均不算完成跨材料对照。
面向用户不得使用“以前的语境”“旧语境”等泛称。使用 `recall.source_label`，明确写成
“材料 A《标题》”“材料 B《标题》”；内部 source_id 不对用户显示。

## 解释方式复用

召回旧知识前先读取 `status` 中的读者状态：

- 过去某种策略已成功时，优先沿用该表示方式；
- 过去某种策略明确失败时，避免用同样结构重复解释；
- 旧例本身曾帮助使用者理解时，优先采用 `old-example-contrast`；
- 没有真实反馈时保持默认最短解释，不虚构偏好。

知识命中与解释策略命中是两件事。旧知识可靠但旧解释失败时，保留旧原句与证据，改用新的解释策略。

## 词汇复用

优先对比核心义和领域义。旧例负责唤醒记忆，新例负责扩展边界。若意义不同，明确标注“同形不同义”，不要错误增加熟悉感。

## 语法复用

用旧句展示结构模板，用新句填入不同成分。重点是让使用者识别机制，而非背诵两条译文。

示例：

```text
旧句：A compiler, which translates source code, ...
新句：The closure, which was returned by the outer function, ...
共同机制：两个逗号之间的 which 从句补充说明前面的名词，不限定它是哪一个。
视觉信号：名词后“逗号 + which”，从句结束后再用逗号回到主干。
```

## 问题复用与旧问题解决

- 相似 QST 出现时，先展示过去原始问法与旧状态，再说明当前问题新增了什么。
- 新材料提供答案时，记录一个同 key 的新 QST occurrence，`status=resolved`，并用 `related_to + relation=resolves` 链接旧事件。
- 新证据只是补充时用 `supports`；改变旧理解时用 `revises`；指出边界时可用 `contrasts`。

## 复用日志

只有回答中实际使用了旧记忆，才运行 `mark-reuse`。召回到但未采用的候选不记成功。这使后续可以区分“检索到了”和“真正帮上忙了”。

## 无命中

无可靠候选时：

1. 明确告诉使用者项目中未找到旧记忆。
2. 正常解释当前问题。
3. 若符合沉淀标准，记录为新记忆。

绝不能为了表现系统有效而强行匹配。
